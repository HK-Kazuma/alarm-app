## 1. アプリ概要

### コンセプト
PS Vitaの「目覚まし同盟」をベースに、**二度寝防止**と**航海テーマ**の世界観を加えたソーシャル目覚ましアプリ。同じ時刻に起きるユーザー同士が「同盟」を組み、仲間を起こし合うことで起床を習慣化する。

### テーマ・世界観
- **アプリテーマ: 航海**
- 同じ時刻に起きるユーザーと同盟（船の乗組員）を組む
- 現実よりは夢の中のイメージ
- 寝坊すれば船が沈没 → 仲間を起こして船に乗せてあげよう

### 従来アプリの問題点（解決すべき課題）
- 寝過ごすと機能が停止する
- 全員が起きるまで応援をし続けないとアプリが閉じれない

## 2. 機能要件

### コア機能
| 機能 | 詳細 |
|---|---|
| アラーム設定 | 起床時刻を設定。同じ時刻に起きるユーザーと自動マッチング |
| マッチング | ユーザーが十分にいない場合はボットで補完 |
| 起床チャレンジ | 目覚ましが鳴ると動き回る自分のアイコンをタップするまで鳴り続ける |
| 応援機能 | 起床後、周りのユーザーをタップして起床を助ける |
| 応援の自動化 | 起床後の応援操作は自動で行える（洗顔・水を飲む等の時間確保） |
| 時計画面選択 | 時計のデザインを選択可能 |

### 二度寝防止・習慣化機能
| 機能 | 詳細 |
|---|---|
| スマホ使用制限 | 起床後スマホを触り続けないよう制限（SNSアプリ制限、ネット一定時間制限など） |
| 起床後タスク表示 | 起床後にやるべきタスクを表示 |

### ゲーミフィケーション
| 機能 | 詳細 |
|---|---|
| 起床記録 | 起床時刻通りに起きた回数、応援した/された回数を記録・表示 |
| トロフィー | 実績に応じてトロフィーを獲得 |
| 航海演出 | 寝坊で船が沈没、仲間を起こして船に乗せる等のアニメーション |

## 3. 技術的な特性と課題

このアプリには技術選定に大きく影響する4つの特殊要件がある。

| # | 要件 | 技術的課題 |
|---|---|---|
| ① | アラームのバックグラウンド動作 | アプリが閉じていても確実にアラームを鳴らす必要がある。OSのネイティブAPIに深く依存 |
| ② | リアルタイム通信 | マッチングや起床タップの同期にWebSocket基盤のリアルタイム通信が必要 |
| ③ | 他アプリの制限（SNS制限） | iOS: Screen Time API、Android: UsageStatsManager。最もネイティブ依存度が高い |
| ④ | リッチなアニメーション | 動き回るアイコン、船の沈没演出など航海テーマのアニメーション |

## 4. 技術スタック（決定）

**React Native + Expo（→ 段階的にDevelopment Buildへ移行）** を採用。

| レイヤー | 技術 | 選定理由 |
|---|---|---|
| フロントエンド | **React Native + Expo** | TypeScript/Reactの既存知識が活かせる |
| 言語 | **TypeScript** | 料理アプリと共通 |
| アラーム | **react-native-alarm-notification** + ネイティブモジュール | バックグラウンドアラーム処理 |
| リアルタイム通信 | **Supabase Realtime** | WebSocket基盤。マッチング・起床通知の同期 |
| バックエンド | **FastAPI（Python）** | 既存スキル活用。マッチング・ボット処理・統計集計 |
| DB | **Supabase（PostgreSQL）** | ユーザー情報、起床記録、トロフィー管理 |
| 認証 | **Supabase Auth** | ソーシャルログインも追加可能 |
| プッシュ通知 | **Firebase Cloud Messaging（FCM）** | 「仲間が寝坊してるぞ！」等の通知 |
| アニメーション | **react-native-reanimated + Lottie** | 航海テーマの船・波・キャラの動き |
| SNS制限 | **ネイティブモジュール自作（Swift / Kotlin）** | OS APIへの直接アクセスが必須 |

### 開発環境の移行戦略

```
Phase 1〜2 : Expo Go で開発（環境構築ゼロ、Hot Reload）
Phase 3〜4 : Expo Development Build（EAS Build）に移行
             → Expo Goの開発体験を維持しつつネイティブモジュール追加可能
             → コードの書き直し不要（設定追加のみ）
```

### 開発環境の比較（検討済み）

| | Expo Go | Dev Build (EAS) | RN CLI | ローカルExpoビルド |
|---|---|---|---|---|
| 環境構築 | ほぼゼロ | 軽い | 重い | 重い |
| Hot Reload | ○ | ○ | ○ | ○ |
| ネイティブモジュール | × | ○ | ○ | ○ |
| Mac不要(Android) | ○ | ○ | △ | × |
| 学習コスト | 低 | 低〜中 | 高 | 中〜高 |
| ストア公開 | × | ○ | ○ | ○ |

→ **Expo Go → Expo Development Build** の2段階移行が最適と判断。

## 5. 開発フェーズ

### Phase 1: 基盤（1人で使える最小版）
- **開発環境**: Expo managed workflow（Expo Go）
- アラーム設定・時計画面選択
- 基本UI（航海テーマ）
- ローカルでの起床記録

### Phase 2: ソーシャル機能
- **開発環境**: Expo Go（継続）
- FastAPIバックエンド構築
- Supabaseでユーザー管理・マッチングロジック
- リアルタイム通信（起床タップの同期）
- ボットユーザーの実装

### Phase 3: ゲーミフィケーション
- **開発環境**: Expo Go → Development Build 移行検討
- トロフィー・実績システム
- 船の沈没/救出アニメーション（Lottie / Reanimated）
- 統計ダッシュボード

### Phase 4: 高度な機能
- **開発環境**: Expo Development Build（EAS Build）
- bare workflowへのeject
- バックグラウンドアラームの強化
- SNS制限（ネイティブモジュール）
- ストアへの公開準備

## 7. リスク・注意事項

### 技術リスク
- **SNS制限機能（iOS）**: Screen Time API（Family Controls / ManagedSettings）はApple Developer Programの特別な承認が必要。ストア審査のハードルが高い
- **SNS制限機能（Android）**: UsageStatsManagerへのアクセスは比較的容易だが、ユーザーに手動で権限許可してもらう必要がある
- **FastAPI無料ホスティングのコールドスタート**: Renderの無料枠はアクセスがないとサーバーがスリープ。早朝のリクエスト集中時に30秒〜1分の遅延が発生する可能性

### MVP戦略
- SNS制限はMVP段階では「起床後のタスク表示」のみにして、ネイティブレベルのアプリ制限は後フェーズに回すのが現実的
- Apple Developer登録のタイミングはPhase 2のテスト段階で判断（iOSでTestFlightを使う場合に必要）
